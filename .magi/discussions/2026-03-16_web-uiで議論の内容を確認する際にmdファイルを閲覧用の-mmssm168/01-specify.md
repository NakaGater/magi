【PD】

⚠️ 懸念: PMの工数優先度判断は理解しますが、ユーザー体験の観点から重要な問題があります。

現在のwhitespace-pre-wrap表示での具体的な問題点：

1. **認知負荷の増大**: マークダウン記法（`##`, `*`, `-`等）がそのまま表示されることで、非エンジニアユーザーは議論内容の理解に余計な解釈コストがかかる

2. **情報階層の不明確**: 見出しレベルが視覚的に区別されないため、議論の構造把握が困難。特にSpec.mdやADR等の構造化文書では致命的

3. **コードと文章の混在**: インラインコードやコードブロックが通常テキストと区別されず、技術的議論での誤解を招く可能性

4. **スキャナビリティの低下**: 長い議論を流し読みする際に、重要ポイント（見出し、リスト項目）を素早く識別できない

代替案として提案されたCSS改善では、マークダウン記法自体は残るため根本的解決になりません。

react-markdownは軽量で実績があり、sanitization機能も内蔵。1-2日の投資で長期的なUX改善が見込める合理的な選択です。

「美しい表示」ではなく「理解しやすい表示」という基本的なユーザビリティ要件として捉えるべきではないでしょうか？

【Dev】

⚠️ 懸念: 実装工数とリスクの観点から複数の技術的課題があります。

**技術リスクの具体的分析:**

1. **バンドルサイズ**: react-markdownは軽量と言われますが、remark/rehypeエコシステム込みで実質30-50KB。既存のWeb UIバンドルに対する影響を測定せずに「50KB以下」の制約は根拠不足

2. **XSS対策の複雑性**: LLM生成コンテンツのサニタイゼーションは想定より複雑。react-markdown単体では不十分で、rehype-sanitize等の追加設定が必要。設定ミスによる脆弱性リスク

3. **パフォーマンス影響**: 長い議論での複数StatementCard同時レンダリング時、マークダウンパースが各コンポーネントで個別実行される。メモ化対応が必須だが設計されていない

4. **既存アーキテクチャとの整合性**: 現在のwhitespace-pre-wrapは意図的選択。LLM出力の改行・インデントを保持する目的があり、マークダウンレンダリングで情報が失われる可能性

**代替提案:**
まず現在の表示問題の定量的な評価（ユーザビリティテスト）を行い、CSS改善（コード部分のハイライト、見出し風スタイル）で解決可能かを検証。

技術的実現は可能ですが、MVP完成前の機能追加としてはリスクが高すぎます。

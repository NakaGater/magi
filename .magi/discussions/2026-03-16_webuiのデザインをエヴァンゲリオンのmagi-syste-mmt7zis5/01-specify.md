⚠️ **懸念：Specify に進む前に、技術的実現可能性と既存機能への影響を明確にすべき**

PM と PD の指摘に同意します。加えて**技術視点から 3 つの具体的リスク**を指摘します：

**1. テスト欠落による UI 改修リスクが極めて高い**

現状：
- Web UI テスト = 0 ケース（StatementCard・StageProgress・ConsensusMarker 未テスト）
- SSE リアルタイム表示、メッセージ入力フォーム、ライブビュー自動スクロール = すべて手動検証のみ

エヴァンゲリオンテーマ化は「現在のカラー・レイアウト・要素配置の大規模改修」を伴います。**テストなしの状態でこれを実施すると、以下のリグレッションが確実に発生します：**
- SSE 接続断時の自動再接続ロジックの破損
- メッセージ入力フォーム送信時のローディング状態管理の混乱
- ラウンド区切りの時系列グループ化表示による React key 衝突

**代替案**：
- Specify を進める**なら**、まず「テスト追加 3-5 日」を前置条件とする
- または、「テーマ化は v1.1 正式版ではなく、実験的ブランチで実施 → ユーザーフィードバック後に main へマージ」という段階的アプローチ

**2. ビルドパフォーマンス・バンドルサイズへの未知の影響**

エヴァンゲリオンテーマ化に「複雑なアニメーション・装飾 CSS」が含まれると想定される場合、以下の測定が必須です：

- **Tailwind CSS クラス数の増加** → バンドルサイズ（現在 ~150KB）が 30-50% 増加する可能性
- **ランタイム JavaScript 増加** → useAnimation Hook 等の状態管理追加時の Next.js ビルド時間（現在 ~60-90 秒）への影響

**このステージで Specify を詳細化してから測定すると、「実装後に計測→パフォーマンス悪化判明→リファクタ」という後戻りが発生します。**

**代替案**：Specify 前に「技術スパイク」を実施
- エヴァンゲリオンテーマの「最小限のビジュアル実装例」を 1-2 日で試作
- バンドルサイズ・ビルド時間を測定
- 許容範囲か判定してから Specify 開始

**3. 既存コード資産（PM/PD/Dev ロール別カラーシステム）との競合**

現在の design-system.md に定義済み：
```
PM: #4A90D9（青）
PD: #E8A838（橙）
Dev: #50C878（緑）
```

エヴァンゲリオンテーマ化で「第 3 新東京市のサイバーパンク配色」を導入した場合、**これらロールカラーが埋没する可能性があります。**

**具体的リスク**：
- StatementCard の左ボーダーカラーが「テーマ全体の配色に溶け込む」→ ロール識別能力が低下
- 合意済み要件の US-1（「3 視点の議論構造が画面レイアウトで自明である」）と矛盾する

**代替案**：
- テーマ化時に「ロール別カラーの視認性を確保するための制約」を Specify に含める
- または、「テーマ化は背景・テクスチャのみで、ロールカラーは現状維持」と限定

---

**総合判定：❌ 却下提案 - 現段階での Specify 進行**

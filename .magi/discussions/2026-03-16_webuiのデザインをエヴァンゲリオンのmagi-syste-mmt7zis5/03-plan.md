⚠️ 懸念: PM・PD の指摘を全面支持します。加えて**技術的リスクと実装負債**の観点から3つ整理します。

**懸念1: テスト環境ゼロでのHTML構造刷新は本番障害確定**

現在の Web UI は SSE 接続・メッセージキュー・React 状態管理の重要機能を持ちながら、ユニットテストが実装されていません。テーマ化は「flex レイアウト→grid」「CSS カスタムプロパティ追加」等の HTML 構造刷新を伴います。テスト基盤なしでこれを実施すると：
- SSE 接続断時の自動再接続ロジック破損
- メッセージ送信時のローディング状態管理の混乱
- React key 衝突による再レンダリングバグ

修正コストが本番後だと3倍以上になります。

**懸念2: 優先度の逆転は後戻り確定**

現在の既知課題優先度：
1. **Markdown フルレンダリング**（基本機能の完全化）
2. **ユニットテスト導入**（本番リリース前の品質基盤）
3. **エラーハンドリング強化**（SSE 再接続ロジック等）
4. **装飾層のテーマ化**（機能完成後の拡張）

現段階でタスク4を先行すると、タスク1-3が圧迫され、MVP 品質が下がります。

**代替案**: 以下の段階的実装スケジュール提案：
- **Week 1-2（MVP リリース）**: Markdown 完全化 + テスト10ケース + アクセシビリティベースライン測定に集中。テーマ化は排除
- **Week 3（v1.1 再評価）**: PM の要望出処確認 + Dev の技術スパイク（配色変更のみ）+ PD のユーザーリサーチ結果から「実装するか判定」

**結論**: 現時点で「エヴァンゲリオンテーマ化タスク」を分割するべきではありません。代わりに以下4つの**前置条件クリアタスク**をタスク化すべき：
1. PM: 要望出処確認（1日）
2. Dev: テスト導入 + アクセシビリティベースライン測定（3-5日）
3. Dev: 技術スパイク実行（配色変更のみ、1日）
4. PD: ユーザーリサーチ計画 + 実施（2-3日）

これらが完了した時点で初めて「テーマ化を実装するか」を定量データに基づいて判定できます。

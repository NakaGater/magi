# 要件定義: Magi Phase1リリース - Specモード特化版

## 概要
現在90%完成のMVPにおいて、Buildモードの複雑な実装を後回しにし、Specモードを完全化してAI3ロール議論による要件定義自動化ツールとして先行リリースする。段階的リリース戦略によりユーザーフィードバックを早期収集し、Phase2でのBuildモード開発に活かす。

## ユーザーストーリー
- US-1: PM/PDとして、AI3ロール議論で見落としのない要件定義を自動生成したい / 受入基準: PM、PD、Devの3つの視点で議論が進行し、ADRと要件定義書が生成される
- US-2: ユーザーとして、過去の議論履歴を確認したい / 受入基準: History画面で過去の議論一覧が表示され、詳細を閲覧できる
- US-3: ユーザーとして、LLM API失敗時にも議論を継続したい / 受入基準: API失敗時にエラーメッセージが表示され、再試行または部分結果保存が選択できる
- US-4: ユーザーとして、Buildモードの提供予定を知りたい / 受入基準: Buildモード選択時に「Coming Soon」画面でPhase2タイムライン(2026年5月予定)が表示される
- US-5: ユーザーとして、議論の途中で介入したい / 受入基準: 議論進行状況が可視化され、介入可能なタイミングが明確に分かる

## 機能要件
- FR-1: SQLiteベースの永続化機能 - サーバー再起動時も議論データが保持される
- FR-2: History機能の完全実装 - `.magi/discussions/`から過去議論を読み込み表示
- FR-3: エラーハンドリング強化 - LLM API失敗、Git操作失敗時のリカバリー機構
- FR-4: Coming Soon UI実装 - Buildモード選択時の期待管理画面
- FR-5: 議論進行状況の可視化 - ラウンド進捗とユーザー介入タイミングの明確化
- FR-6: 部分結果保存機能 - API失敗時も議論の途中状態を保持
- FR-7: 生成物プレビュー機能 - 要件定義書、ADR、Mermaidダイアグラムの表示

## 非機能要件
- NFR-1: 可用性 - サーバー再起動時のデータ消失ゼロ(SQLite永続化)
- NFR-2: 応答性 - LLM API失敗時も5秒以内にエラー通知
- NFR-3: 拡張性 - Phase2でのスキーマ変更に対応可能な設計
- NFR-4: ユーザビリティ - エラー発生時も議論継続可能な設計
- NFR-5: 透明性 - 機能制限(Buildモード未実装)の明確な伝達

## スコープ外
- LSP統合機能
- 完全なコード生成機能
- GitHub Projects連携(Syncステージ)
- 複雑なファイル操作・構文解析
- vscode-languageserver-protocolの実装

## 前提条件
- 現在のSpecモード基本機能が動作していること
- Git操作の基本機能が実装されていること
- LLM APIとの基本的な連携が確立していること
- Phase2リリース予定は2026年5月として設定

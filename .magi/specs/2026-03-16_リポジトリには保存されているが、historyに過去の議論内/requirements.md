# 要件定義: History機能の実装

## 概要
リポジトリに保存されている過去の議論内容がHistoryに表示されない問題を解決し、ユーザーが過去の議論を振り返れるようにする。データは.magi/discussions/配下に存在するが、読み込み処理が未実装のため空の画面が表示されている状況を修正する。

## ユーザーストーリー
- US-1: ユーザーとして、過去に実行した議論の一覧を時系列で確認したい / 受入基準: History画面で過去の議論がタイトル・日付・モード・ステータス付きで最新順に表示される
- US-2: ユーザーとして、Historyが空の場合やエラーの場合に適切な状態を把握したい / 受入基準: 「議論がまだありません」「読み込み中」「エラーが発生しました」の3状態が明確に区別して表示される
- US-3: ユーザーとして、CLI経由でも過去の議論一覧を確認したい / 受入基準: `magi history`コマンドで基本的な議論一覧が表示される

## 機能要件
- FR-1: サーバー側で`/api/history`エンドポイントが.magi/discussions/配下のデータを読み込み、JSON形式で返却する
- FR-2: 各議論項目にid・タイトル・作成日時・モード（spec/build）・ステータスの基本4フィールドを含む
- FR-3: Web UIでHistory一覧ページに議論項目を最新順（作成日時降順）で表示する
- FR-4: Web UIで読み込み状態・エラー状態・空状態の3パターンを適切に表示する
- FR-5: CLI `magi history`コマンドで基本的な議論一覧を表示する

## 非機能要件
- NFR-1: セキュリティ: ファイルパス走査攻撃を防ぐため、フォルダ名に`..`を含む場合は除外する
- NFR-2: エラーハンドリング: YAMLパース失敗や個別ファイル読み込み失敗は全体処理を停止させず、該当項目のみスキップする
- NFR-3: アクセシビリティ: History一覧でキーボードナビゲーション対応し、各項目に適切なaria-labelを設定する
- NFR-4: パフォーマンス: Promise.allSettledを使用して並列処理でファイル読み込みを効率化する
- NFR-5: 保守性: ContextLoaderの既存ロジックパターンを踏襲し、DRY原則を維持する

## スコープ外
- 議論内容の検索機能
- フィルタリング機能（モード別、ステータス別など）
- 議論項目の詳細プレビュー
- 議論項目の削除・編集機能

## 前提条件
- .magi/discussions/配下に議論データが適切な形式（meta.yaml含む）で保存済み
- 既存のContextLoaderロジックが正常動作している
- Web UIのHistory画面の基本レイアウトが実装済み

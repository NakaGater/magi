# 要件定義: History機能の実装

## 概要
保存された議論データの読み込み処理を実装し、ユーザーが過去の議論を適切に確認できる状態を提供する。

## ユーザーストーリー
- US-1: ユーザーとして、過去の議論一覧を時系列で確認したい / 受入基準: Web UIとCLIで議論一覧が新しい順で表示される
- US-2: ユーザーとして、各議論の基本情報（タイトル、日付、モード、ステータス）を一覧で把握したい / 受入基準: 一覧画面で議論の識別に必要な情報が表示される
- US-3: ユーザーとして、システムの状態（読み込み中、エラー、空状態）を理解したい / 受入基準: 各状態が明確に区別して表示される
- US-4: ユーザーとして、データエラーが発生した場合にその旨を知りたい / 受入基準: 破損ファイルの存在が通知される

## 機能要件
- FR-1: .magi/discussions/配下のmeta.yamlを読み込み、議論の基本情報を取得する
- FR-2: 議論一覧を作成日時の降順でソートして表示する
- FR-3: 各議論について、id・タイトル・作成日時・モード・ステータスの情報を表示する
- FR-4: サーバー側で`/api/history`エンドポイントを実装する
- FR-5: Web UIでHistory一覧画面に議論データを表示する
- FR-6: CLI `magi history`コマンドで基本的な議論一覧を表示する
- FR-7: 読み込み中・エラー・空状態を明確に区別して表示する
- FR-8: 破損したmeta.yamlファイルの件数を通知する
- FR-9: 空状態の場合にユーザーへの誘導メッセージを表示する

## 非機能要件
- NFR-1: 既存のContextLoaderパターンを流用し、コードの一貫性を保つ
- NFR-2: 既存のpath.joinパターンでファイルアクセスのセキュリティを確保する
- NFR-3: 数十ファイル規模での適切なパフォーマンスを確保する
- NFR-4: taskSummaryフィールド未対応の既存データとの後方互換性を保つ
- NFR-5: 最小限のE2Eテスト（ファイル存在時/不存在時）でリグレッションを防止する

## スコープ外
- index.jsonによるインデックス機構
- Promise.allSettledによる複雑な並列処理
- 厳重なセキュリティチェック（path.resolve、boundary check、null byte injection対策）
- アクセシビリティ対応（キーボードナビゲーション、aria-label）
- 検索・フィルタ機能
- 議論内容の要約表示
- ページネーション機能

## 前提条件
- .magi/discussions/配下にmeta.yamlファイルが存在する
- 既存のContextLoader実装が利用可能である
- Web UIとCLIの基本的なインフラが整備されている

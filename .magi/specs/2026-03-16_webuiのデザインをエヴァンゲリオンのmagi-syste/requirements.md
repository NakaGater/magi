# 要件定義: WebUI のデザイン改善 (段階的実装)

## 概要
エヴァンゲリオン MAGI システムをモチーフにした Web UI テーマ化は、技術的基盤（テスト・アクセシビリティ）とビジネス価値（要望出処・ユーザーニーズ）の検証を前置条件として、v1.1 フェーズで段階的に実装する。MVP リリース前は基本機能の完成度と品質基盤の確立に注力する。

## MVP フェーズ (Week 1-2) - 優先度: P0

### ユーザーストーリー
- **US-1**: エンジニア（PM・PD・Dev）として、セッション中に 3 視点の議論構造が一目瞭然に見える必要がある（現在のロール別カラー・シンプルカード配置を維持）
  - 受入基準: ロール別カラー（PM 青・PD 橙・Dev 緑）が背景に埋没せず、初見で識別可能であること

- **US-2**: 営業・採用等の非エンジニアとして、Markdown フォーマットの仕様書（見出しサイズ・リスト・コードブロック）が正確に読める必要がある
  - 受入基準: h2-h4 見出し、リスト（順序付き・無順序）、コードブロックが正確に HTML レンダリングされること

- **US-3**: ユーザーとして、SSE ストリーム接続の断時に自動再接続されることを確認したい
  - 受入基準: ユニットテスト で SSE 再接続ロジックの正常動作が証明されること

### 機能要件

#### Markdown フルレンダリング
- **FR-1**: Markdown ブロックレベルから インラインレベルへの拡張
  - `remark` + `rehype` を導入し、見出し（h2-h4）、リスト（ul/ol）、コードブロックをサーバー側で HTML に変換
  - 現在の dim 処理（参考発言の視覚的軽減）は維持
  - 成果物が正確な Markdown として描画されること

#### ユニットテスト導入
- **FR-2**: React コンポーネントの基本動作検証
  - StatementCard（5 ケース）: userMessage 表示、Markdown dim 処理、ロール別 CSS 適用
  - StageProgress（3 ケース）: waiting/active/done 状態遷移
  - ConsensusMarker（2 ケース）: agreed/partial/disagreed 表示
  - Vitest + @testing-library/react で実装、CI パイプライン統合

- **FR-3**: SSE 接続・メッセージ送信の手動テスト検証
  - 接続断時の自動再接続動作確認
  - メッセージ送信時のローディング状態確認
  - ラウンド進捗のリアルタイム更新確認

### 非機能要件
- **NFR-1**: アクセシビリティ（WCAG 2.1 AA）の現在値を測定・記録
  - Lighthouse Accessibility スコア（現在値ベースライン）
  - axe DevTools による WCAG 項目の自動チェック
  - スクリーンリーダー（NVDA）の読上テスト実施
  - 成果物: `.magi/context/accessibility-baseline.md`

- **NFR-2**: CI 実行時間の増加を +30 秒以内に抑制
  - 現在: 120 秒 → テスト追加後: 150 秒以内

- **NFR-3**: 本番リリース品質の確保
  - ユニットテスト全パス + 手動回帰テスト完了状態でのデプロイ
  - Buildモード出力の正確性確認

### スコープ外（MVP フェーズでは実施しない）
- ❌ エヴァンゲリオン MAGI テーマ化の詳細仕様書
- ❌ カラーパレット刷新（MAGI 配色の設計）
- ❌ UI 要素追加（アイコン・テクスチャ・アニメーション）
- ❌ テーマ化実装のコード変更

### 前提条件
- Web UI が現在 95% 完成の状態で、基本機能（ライブビュー・メッセージ入力・Consensus 判定）が正常動作していること
- GitHub Actions CI/CD パイプラインが構築済みであること
- Tailwind CSS v4、Next.js 14 の現在の依存関係が安定している状態

## v1.1 フェーズ (MVP + 1-3 週) - 優先度: P1 (条件付き)

### ユーザーストーリー
- **US-4**: エンジニアおよび非技術ユーザーとして、MAGI システムの「3 賢者の議論構造」が視覚的に強調され、ツール利用時の心理的満足度が向上する
  - 受入基準: ユーザーリサーチで「テーマ理解度」が 70% 以上、「ハードル評価」が「変わらない/下がった」で 80% 以上であること

- **US-5**: 採用・営業チームとして、テーマ化により初期接触時の「ツール印象」が向上し、ユーザーの継続利用意思が増加する
  - 受入基準: テーマ化前後で NPS スコアが +5 ポイント以上向上、または判定までの所要時間が 10% 以上短縮されること

### 機能要件（v1.1 再評価後に確定）

#### エヴァンゲリオン MAGI テーマ化（条件付き実装）
- **FR-4**: ロール別カラーの視認性維持の下での視覚強調
  - MAGI の 3 視点カラー（紫・赤等）を背景・ボーダー・アクセント層に限定
  - ロール別カラー（PM 青・PD 橙・Dev 緑）の埋没なし
  - 色覚異常ユーザーが Coblis/Daltonize シミュレータでも識別可能な配色設計

- **FR-5**: 軽微なテクスチャ・視覚調整の導入（アニメーション除外）
  - SVG グラデーション による背景テクスチャ（PNG 画像でのバンドルサイズ増加なし）
  - border-width / letter-spacing の調整によるカード読みやすさ向上
  - リアルタイムアニメーション禁止（WCAG 光感受性基準対応）

### 非機能要件
- **NFR-4**: アクセシビリティの悪化なし（WCAG 2.1 AA 準拠を維持）
  - テーマ化前後で Lighthouse Accessibility スコアが低下しないこと
  - スクリーンリーダー読上順序が変わらないこと
  - ハイコントラスト表示での視認性が低下しないこと

- **NFR-5**: パフォーマンスの悪化を +10% 以内に抑制
  - ビルド時間: +10 秒以内
  - バンドルサイズ: +20% 以内
  - Core Web Vitals (LCP/FID/CLS) が 200ms 以内に維持されること

- **NFR-6**: テーマ化に伴う HTML 構造刷新のリグレッション防止
  - 既存の 12 ユニットテスト全パス（WCAG 含む）
  - 手動回帰テスト（SSE・メッセージ送信・ラウンド進捗）完了

### スコープ外（v1.1 でも実施しない）
- ❌ 3Hz 以上のフラッシングアニメーション
- ❌ 色のみで情報伝達（テキスト併用必須）
- ❌ アニメーション CSS による大幅な HTML 構造変更
- ❌ 非エンジニア向けオンボーディング UI への適用（別フェーズで検討）

### 前提条件（全て満たす必要）
1. **PM による要望出処確認＋ビジネス成功指標の定義**
   - 要望出処: 営業・採用・CEO のいずれか（確定）
   - 成功指標: NPS・判定時間短縮率・採用効果等（定量化）
   - 期日: いつまでに必要か（明確化）

2. **Dev による技術検証完了**
   - ユニットテスト 12 ケース全パス（WCAG コントラスト検証含む）
   - アクセシビリティベースライン測定済み（Lighthouse + axe + NVDA）
   - 技術スパイク完了（配色変更のみ実装、ビルド時間・バンドルサイズ測定）

3. **PD によるユーザーリサーチ完了**
   - 非エンジニアユーザー 5-10 名へのインタビュー実施
   - テーマ理解度・ハードル評価・継続利用意思を測定
   - 結果: `.magi/context/user-research-report.md` に記載

4. **設計制約の明確化**
   - `.magi/context/evangelion-scope-constraints.md` で実装 OK/NG 項目を明示
   - ロール別カラー埋没なし、アクセシビリティ制約を明記

## 判定フロー（v1.1 再評価時）

### 実装判定の条件
```
全て満たす場合 → 実装 Specify 開始
  ✅ PM が要望出処・成功指標を確定
  ✅ Dev がテスト 12 ケース + スパイク測定完了
  ✅ PD がユーザーリサーチで「ハードル上昇なし」を定性確認
  ✅ ビルド時間 +10 秒以内、バンドルサイズ +20% 以内

1つ以上不満 → 優先度再調整または非実装判定
```

### 非実装判定の場合
- エヴァンゲリオンテーマ化の要望は棄却
- 代替案: StatementCard ボーダー視認性向上、ラウンド区切り表示改善など、CSS のみの軽微改善に限定

# 要件定義: エヴァンゲリオン MAGI 視覚化テーマの段階的検討

## 概要
WebUI のデザインをエヴァンゲリオンの MAGI システムのテーマに近づけたいという要望に対して、MVP 完成後の v1.1 フェーズで段階的に検討することが合意された。現在の MVP フェーズでは Markdown フルレンダリング完成とテスト導入を優先し、技術的基盤の安定化を図ったうえで、ユーザーニーズの検証を経て実装判定を行う。

## ユーザーストーリー

| ID | ユーザーストーリー | 受入基準 |
|---|---|---|
| US-1 | エンジニア向けツールとして、MAGI のテーマ視覚化により議論プロセスの複雑性を直感的に理解したい | 3 視点の議論構造が画面レイアウトで自明である |
| US-2 | 非エンジニア向けユーザーとして、テーマ化によってツール利用のハードルが上がらないことを確認したい | アクセシビリティ（スクリーンリーダー・ハイコントラスト表示）が損なわれていない |
| US-3 | ステークホルダー（営業・採用・CEO）として、Magi ブランドの差別化要素が強化されたことを確認したい | 対象ユーザーセグメントでのブランド認知度向上を測定可能である |

## 機能要件

### MVP フェーズ（現在の優先度）

| ID | 要件 | 説明 | 実装者 |
|---|---|---|---|
| FR-1 | Markdown フルレンダリング完成 | 見出し（h1-h6）・リスト（順序付き/順序なし）・コードブロック・テーブル等の完全な HTML 対応を実装（`remark` + `rehype` 導入） | Dev |
| FR-2 | ユニットテスト導入（最小化） | StatementCard・StageProgress・ConsensusMarker の計 10 ケースのテスト実装（Vitest + React Testing Library） | Dev |
| FR-3 | エラーバウンダリ強化 | SSE 切断時の再接続挙動検証、ネットワークエラー時のユーザーフィードバック改善 | Dev |
| FR-4 | 既存 UI の視認性微調整（装飾なし） | StatementCard の左ボーダー幅・色コントラスト向上、ラウンド区切りの時系列グループ化表示（HTML 再編成のみ） | Dev/PD |

### v1.1 フェーズ（延期・条件付き）

| ID | 要件 | 説明 | 実装条件 |
|---|---|---|---|
| FR-5 | エヴァンゲリオン MAGI テーマ視覚化（装飾レイヤー） | カラーパレット刷新・UI 要素追加・アニメーション導入等、テーマ性を高めるデザイン実装 | ユーザーニーズ検証 + 要望出処確認完了後 |
| FR-6 | ユーザーインタビュー＆フィードバック収集 | 実ユーザー 5-10 名へのインタビュー、非エンジニア向けオンボーディングユーザーの行動観察 | MVP リリース + 1 週間時点 |
| FR-7 | アクセシビリティ検証（v1.1 向け） | スクリーンリーダー対応・ハイコントラスト表示・キーボード操作の検証（テーマ化による悪化チェック） | テーマ実装前の必須検証 |

## 非機能要件

| ID | 要件 | 説明 | 測定方法 |
|---|---|---|---|
| NFR-1 | ビルドパフォーマンス | Markdown レンダリング導入による Next.js ビルド時間増加を 5-10 秒以内に抑制 | CI/CD ビルド時間ログ比較 |
| NFR-2 | テスト実行時間 | テストスイート導入による GitHub Actions 実行時間を 3 倍以内に収定 | CI 実行時間の測定 |
| NFR-3 | ユーザー操作応答性 | テーマ化実装後も、UI インタラクション（入力・送信・表示）の遅延が 200ms 以下 | 画面キャプチャ・ユーザー操作ログ |
| NFR-4 | アクセシビリティスコア | WCAG 2.1 AA 準拠（現在・テーマ化後両方） | axe DevTools・Lighthouse 測定 |

## スコープ外

### MVP フェーズで実装しないもの
- ❌ エヴァンゲリオンテーマのカラーパレット導入
- ❌ MAGI 特有の UI 要素追加（パイロット表示等）
- ❌ テーマ固有のアニメーション・装飾
- ❌ 全コンポーネントの 100% テストカバレッジ（対象は 3 コンポーネント最小化）
- ❌ Buildモード完全実装試験（デザインステージまでの試験運用のみ）

### v1.1 フェーズの検討対象外
- ❌ ライト/ダーク複数テーマの切り替え機能（ダークテーマ継続）
- ❌ リアルタイムアニメーション（静的ビジュアル強化のみ）
- ❌ モバイルレスポンシブ対応の再設計（現在の web パッケージは PC 向け）

- [ ] StatementCard・StageProgress・ConsensusMarker のユニットテスト 10 ケース完成
- [ ] 技術スパイク検証完了：ビルド時間・テスト実行時間が許容値内であることを確認
- [ ] エラーバウンダリ・SSE 再接続ロジックの動作確認
- [ ] 現在の 95% 完成状態のコンポーネントで回帰テスト実施

### v1.1 再評価の前提
- [ ] MVP が本番環境で 1 週間安定稼働を確認
- [ ] **要望出処の明確化**：営業・採用・CEO・ユーザーのいずれからの要望かを PM が確認（ステークホルダー名・背景・期日）
- [ ] ユーザーフィードバック収集（実ユーザー 5-10 名のインタビュー実施）

  - 非エンジニア向けオンボーディング の理解度向上
- [ ] 受入基準の設定：成功をどのメトリクス（NPS・操作時間・学習時間等）で測定するかを決定

## 重要な制約・リスク

### 技術的リスク
- **リグレッション懸念**：テストゼロの状態での大規模 UI 改修は既存機能（ライブビュー・メッセージ入力・SSE 接続）を破損する可能性が高い
- **ビルド影響**：Markdown レンダリング導入によるバンドルサイズ・実行時間増加は未測定
- **テスト環境コスト**：CI 時間が大幅増加する可能性

### ビジネス的リスク
- **要望出処の不明確性**：現在、誰がこの要望を出したのか、背景は何なのかが定義されていない
  - v1.1 で再検討する際に「実は営業から 2 週間前のリクエストだった」等、優先度が覆る可能性あり
- **アクセシビリティ悪化懸念**：視覚的装飾を優先すると、スクリーンリーダー対応やハイコントラスト表示に対応しにくくなる
- **ユーザー認知の障壁**：エヴァンゲリオンテーマは科学者・エンジニア向けなどの一部ユーザーに理解されない可能性

### プロセス的リスク
- **「延期 = 先送り」化のリスク**：v1.1 で改めて議論になり、結局実装されない可能性
  - 対策：本ドキュメント（ADR）の定期レビュー、ユーザーリサーチの早期実施

## 意思決定プロセス

### MVP 完成チェックリスト（今週中に確定）
1. **Markdown レンダリング完全化** → スコープ：remark + rehype での見出し・リスト・コードブロック対応（完了基準：Buildモードコード出力の見出しが正確に表示）
2. **テスト追加（最小化）** → 対象：StatementCard（5 テスト）+ StageProgress（3 テスト）+ ConsensusMarker（2 テスト）= 計 10 ケース
3. **技術スパイク実施** → ビルド時間・テスト CI 実行時間を測定、2-3 週での MVP 完成実現可能性を検証

### v1.1 再評価トリガー
- **タイミング**：MVP リリース + 1 週間（ユーザーフィードバック 初期集計時点）
- **実施項目**：
  1. 要望出処の確認（PM が営業・採用等のステークホルダーに再確認）
  2. ユーザーインタビュー実施（非エンジニア・新規ユーザー中心）
  3. ビジネス価値・受入基準の定義
  4. アクセシビリティ影響評価
- **判定結果**：実装する / しない / 優先度再調整

## 成功基準（MVP フェーズ）

- ✅ Markdown フルレンダリング完成：Buildモードで出力されるコード例（見出し・リスト・テーブル）が正確に表示される
- ✅ テスト導入完成：10 ケースのテストが CI で自動実行され、全パス状態を維持
- ✅ 本番稼働安定性：MVP リリース後 1 週間で回帰テスト無しで既存機能の正常動作を確認
- ✅ ビルドパフォーマンス許容：CI ビルド時間が 5-10 秒の増加に収定

## 成功基準（v1.1 フェーズ・条件付き）

- ✅ ユーザーニーズ検証完了：5-10 名のユーザーインタビューで「テーマ化の価値」を定量・定性評価
- ✅ ビジネス価値が定義される：営業・採用等のステークホルダーからの要望背景が明確化
- ✅ アクセシビリティ保証：テーマ化後も WCAG 2.1 AA 準拠を実証
- ✅ 受入基準満足：決定された測定指標（判断時間短縮率・NPS・学習時間等）で目標達成

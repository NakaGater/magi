# ADR-002: MVP リリース前のテスト導入戦略

## ステータス
承認済み

## コンテキスト
現在、SSE ストリーム処理・メッセージキュー・React 状態管理が全て手動テスト頼みで、ユニットテストが実装されていない。本番リリース1-2週間前のこのタイミングで、大規模なUI改修（テーマ化等）の提案が複数出ている。テスト環境ゼロの状態でUI刷新を実施した場合、本番でのSSE接続断・メッセージ送信失敗などの重大リグレッション検出が不可能。

## 決定
MVP リリース前に以下のテスト基盤を導入し、本番デプロイ前に「重大なリグレッション80%検出」を目指す：

**Week 1 実装スケジュール**：
- Mon-Tue: StatementCard（5ケース）・StageProgress（3ケース）・ConsensusMarker（2ケース）のユニットテスト実装（Vitest + @testing-library/react）
- Wed: WCAG カラーコントラスト検証テスト（2ケース）追加、CI パイプライン構築
- Thu: 手動テスト実施（SSE接続・メッセージ送信・ラウンド進捗）+ NVDA スクリーンリーダー実機検証
- Fri: 技術スパイク実行（配色変更のみ）→ ビルド時間・バンドルサイズ測定

**成果物配置**：
- テストコード：`packages/web/src/components/__tests__/`（12ファイル）
- CI 設定：GitHub Actions に自動テスト実行パイプライン追加
- 測定報告書：`.magi/context/accessibility-baseline.md`

## 理由
- **高い投資対効果**：テスト10ケース実装（4-5時間）により、SSE接続断・メッセージ送信失敗といった本番で発見困難な障害を事前検出可能。修正コストが本番後だと3倍以上になる
- **大規模UI改修時のリグレッションリスク低減**：テーマ化による HTML構造変更（flex→grid）と CSS カスタムプロパティ追加時に、自動テストがあれば無責任な変更を防ぎ、重大機能のリグレッション確率を80%から20%以下に低減
- **アクセシビリティの品質保証基盤**：WCAG テスト + NVDA 実機検証により、テーマ化時の色覚異常・スクリーンリーダー互換性の悪化リスクを事前に検出
- **本番リリースの信頼性向上**：テスト全パス + 手動回帰テスト完了の状態でリリースすることで、ユーザーの信頼を損なわない品質を確保

## 却下した代替案
- **テスト導入をv1.1以降に延期**: 本番環境で初めてテストが入ると、リグレッション検出が遅延。既にリリース済みの機能が壊れている状態から修正開始になり、ユーザー信頼喪失
- **100%テストカバレッジ達成を目指す**: 時間効率が悪い。「重大なリグレッション検出」に絞った10ケースで十分

## 影響
- **開発スケジュール**：Week 1 に5-6時間のテスト実装工数が必須。ただし、エヴァテーマ化の実装を延期することで相殺可能
- **CI 実行時間**：現在120秒から150秒以内に増加（+30秒以内に抑制）。本番デプロイメント速度への実質的な悪影響なし
- **将来のUI改修時の生産性**：テスト基盤が確立することで、v1.1以降の機能追加・テーマ化実装時の安全性と速度が向上
- **ビジネスリスク軽減**：本番障害率が大幅に低減され、採用・営業チームへの信頼性向上に直結

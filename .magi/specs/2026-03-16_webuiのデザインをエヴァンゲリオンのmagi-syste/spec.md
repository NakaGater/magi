# 機能仕様書: エヴァンゲリオン MAGI テーマ化（v1.1 延期判定）

## 概要

本仕様書は、WebUI デザインを エヴァンゲリオンの MAGI システムテーマに近づける要望に対する**延期判定ドキュメント**です。

当初「MVP 完成後の v1.1 フェーズで段階的検討」として合意された要件に対し、以下の理由により**詳細仕様化を保留**します：

- **要望出処の不明確性**：営業・採用・CEO のいずれからの要望か確認されていない
- **ユーザーニーズの未検証**：非エンジニアユーザーへの学習曲線への影響を測定していない
- **技術可能性の未測定**：テスト環境がゼロの状態での大規模 UI 改修リスクが評価されていない

本ドキュメントは、以下の**前置条件クリア後に改めて Specify ステージを起動**するまでの暫定記録です。

---

## 判定基準と延期理由

### 判定内容

| 項目 | 状態 | 根拠 |
|---|---|---|
| **要望出処確認** | ❌ 未完了 | PM が営業・採用・CEO のいずれからの要望かを未確認 |
| **ユーザーニーズ検証** | ❌ 未完了 | PD のユーザーインタビュー（非エンジニア向け）が未実施 |
| **技術実現可能性測定** | ❌ 未完了 | Dev のテスト導入・スパイク実施がまだ |
| **アクセシビリティベースライン** | ❌ 未測定 | MVP 時点で WCAG 2.1 AA 準拠度を測定していない |

### 延期判定の根拠

#### 1. リグレッションリスク（Dev 指摘）

現在の Web UI はテスト 0 ケースの状態です。エヴァンゲリオンテーマ化は「カラーパレット・レイアウト・UI 要素配置の大規模改修」を伴うため、以下のリスクが極めて高い：

- SSE 接続断時の自動再接続ロジック破損
- メッセージ入力フォーム送信時のローディング状態管理混乱
- ラウンド区切りの時系列グループ化表示による React key 衝突

**テスト導入なしの状態で Specify を進めると、実装後のリグレッション修正コストが 3 倍になります。**

#### 2. ビジネス価値の不定義（PM 指摘）

要望背景が不明確なため、成功指標が定義されていない：

- 営業からの要望なら：採用ブランド認知度向上を測定すべき
- 採用からの要望なら：候補者の初期スクリーン時間短縮を測定すべき
- CEO からの要望なら：全社コミュニケーション効率化を測定すべき

**不明確な要望に対して仕様化を進めると、v1.1 で「実は 2 週間以内に必要だった」と言われた場合、全て再計画になります。**

#### 3. ユーザー体験の検証漏れ（PD 指摘）

合意済み要件 US-2「非エンジニア向けユーザーとして、テーマ化によってツール利用のハードルが上がらない」が机上の空論のままです：

- エヴァンゲリオンテーマは「エンジニア・SF 好きには刺さるが、営業・非技術者には理解されない」可能性が高い
- スクリーンリーダー対応・ハイコントラスト表示への影響が未測定
- 現在のロール別カラーシステム（PM：青、PD：橙、Dev：緑）が埋没する懸念

**ユーザーリサーチなしで仕様化すると、実装後に「やはり使いづらい」という後戻りが確定します。**

## 画面仕様

### 注記：本セクションは保留状態です

以下の画面仕様は、**前置条件クリア後の v1.1 Specify ステージで初めて詳細化されるもの**です。現段階では「想定される変更範囲」を概略示します。

### [画面名] StatementCard（テーマ化後の想定変更）

#### 目的
議論ステージ内の意見カード表示。現在のロール別カラー識別に加えて、エヴァンゲリオン MAGI テーマの視覚要素を統合することを検討。

#### 構成要素（現在）
- 左ボーダー：ロール別カラー（PM 青 / PD 橙 / Dev 緑）
- 背景：無地白（ダークテーマ対応）
- テキスト：Markdown HTML レンダリング
- メタデータ：作成者・タイムスタンプ

#### 想定される変更要素（v1.1 検討対象）
- ⚠️ カラーパレット刷新（ただし、ロール別色の視認性は維持）
- ⚠️ 背景テクスチャ追加の可能性（ただし、テキスト可読性を損なわない）
- ⚠️ ボーダースタイル変更の可能性（3D インセット等）
- ❌ アニメーション効果は現段階では計画なし

#### 振る舞い（変更なし）
- クリック時の詳細表示（変更予定なし）
- メッセージ入力フォームとの連動（変更予定なし）
- SSE リアルタイム更新（変更予定なし）

#### 制約条件
- WCAG 2.1 AA 準拠を維持（テーマ化後も）
- スクリーンリーダー読み上げ可能性を損なわない
- ロール別カラーが視認可能であること（US-1 「3 視点が自明」を実現するため）

### [画面名] StageProgress（テーマ化後の想定変更）

#### 目的
ディスカッションの進行状況（ラウンド進捗）を表示。MAGI の複数視点統合概念との連携表示を検討。

#### 構成要素（現在）
- プログレスバー：線形表示（現在のステージ位置を示す）
- ステージ標識：ラベルテキスト
- 時系列グループ化：ラウンド別の区切り

#### 想定される変更要素（v1.1 検討対象）
- ⚠️ ビジュアル階層化（ラウンド区分の視覚強調）
- ⚠️ アイコン追加の可能性（各ステージを象徴するビジュアル）
- ❌ リアルタイムアニメーションは計画なし

#### 振る舞い（変更予定なし）
- ステージクリック時のナビゲーション（変更なし）
- 進捗計算ロジック（変更なし）
- SSE によるリアルタイム更新（変更なし）

#### 制約条件
- ビルド時間増加を +10 秒以内に抑制
- Core Web Vitals 200ms レスポンス遅延なし

### [画面名] ConsensusMarker（テーマ化後の想定変更）

#### 目的
ステートメントの合意度・投票状況を表示。MAGI の 3 視点投票（PM/PD/Dev）概念を強化するビジュアル検討。

#### 構成要素（現在）
- 投票ボタン：賛成 / 反対 / 棄権
- 投票結果表示：集計数字
- メーター表示：合意率（0-100%）

#### 想定される変更要素（v1.1 検討対象）
- ⚠️ ロール別投票者の視覚的区別（色分け）
- ⚠️ メーター表示の強調（テーマ色採用）
- ❌ アニメーション効果は計画なし

#### 振る舞い（変更予定なし）
- 投票ロジック（変更なし）
- SSE リアルタイム集計更新（変更なし）

#### 制約条件
- ロール別カラーの埋没防止（テーマ色が濃すぎない）
- アクセシビリティ：色覚異常ユーザーへの配慮（色分けだけでなくテキスト表示も併用）

## API 仕様

### 注記：本セクションは変更なし

エヴァンゲリオンテーマ化は **UI/UX レイヤーの改修**のため、API レベルの変更は想定されません。以下は現在の API 構成です。

### [エンドポイント] GET /api/discussions/{discussionId}

#### メソッド
GET

#### 目的
指定ディスカッション ID の全体データ（ステートメント・投票・ステージ進捗）を取得

#### リクエスト
```
GET /api/discussions/{discussionId}

パラメータ:
  discussionId (string): ディスカッション ID（パスパラメータ）
  fields (array, optional): 取得フィールド指定（省略時は全て）
    例: fields=statements,stage,consensus
```

# ADR-003: アクセシビリティ検証の前置条件化

## ステータス
承認済み

## コンテキスト
エヴァンゲリオンテーマ化により、MAGI の3視点カラー（紫・赤等）が導入される。現在のロール別カラー（PM青・PD橙・Dev緑）がテーマ色に埋没する懸念、および色覚異常・視覚障害ユーザーがテーマ装飾により排除される可能性がある。特に非エンジニアユーザー向けのオンボーディング機能が将来実装される際、アクセシビリティの不備は採用・営業効果に直結する負の影響となる。

## 決定
エヴァンゲリオンテーマ化を Specify ステージに進める前に、以下のアクセシビリティ検証を必須前置条件とする：

**Week 1 実装内容**：
1. **Lighthouse + axe DevTools による自動測定**（Wed）
   - Accessibility スコア記録（現在値ベースライン）
   - WCAG 2.1 AA 準拠項目の自動検証
   - 成果物：`.magi/context/accessibility-baseline.md`

2. **スクリーンリーダー実機テスト**（Thu）
   - NVDA（Windows）による読上テスト実施
   - DOM構造の読上順序・属性（role, aria-label）の正確性確認
   - 非エンジニアの視点で「機能名・ボタン機能が正確に伝わるか」検証

3. **ハイコントラスト表示モード検証**（Thu-Fri）
   - Windows ハイコントラストモード下での視認性確認
   - 色覚異常シミュレータ（Coblis / Daltonize）による配色テスト
   - ロール別カラーが色盲ユーザーでも識別可能か確認

4. **スコープ・制約ドキュメント作成**（Fri）
   - テーマ化で実装 OK な項目（カラーパレット・軽微テクスチャ）と NG な項目（3Hz以上フラッシング・色のみ情報伝達）を明示
   - 成果物：`.magi/context/evangelion-scope-constraints.md`

## 理由
- **ユーザー排除の防止**：色覚異常ユーザー（日本人男性の約8%）、視覚障害ユーザー（スクリーンリーダー利用者）がテーマ化により実質的に排除されることを事前検出可能
- **採用・営業効果への直結**：非エンジニアユーザーが利用する将来のオンボーディング機能が、テーマ化によってハードル上昇する可能性を定性データで検証
- **後からの修正不可能性**：アクセシビリティの不備は「信用損失」につながり、実装後の修正は組織の信頼回復に多大なコストがかかる
- **テーマ化実装時の設計ガイド化**：事前に「OK / NG」の制約を明示することで、実装段階での無責任な判断（「見た目がカッコいいからアニメーション追加」等）を防止

## 却下した代替案
- **テーマ化実装後のアクセシビリティ修正**: アクセシビリティ不備は後から修正が困難。特に「色だけで情報伝達」といった設計レベルの問題は実装後の修正では対応不可能
- **自動ツール検証のみで実機テスト省略**: axe DevTools は自動検証の限界があり、スクリーンリーダーの読上順序・ハイコントラスト下の実際の視認性は実機テストなしに判定不可

## 影響
- **テーマ化実装の前提条件化**：v1.1 での実装判定が「ビジネス価値 + 技術実現性 + アクセシビリティ確保」の3軸で行われ、品質基準が明確化
- **ユーザーニーズ検証との統合**：非エンジニアユーザー向けのユーザーリサーチ実施時に、テーマ理解度・ハードル評価・アクセシビリティ自己評価を同時測定可能
- **将来のUI改修基準化**：「テーマ化 = アクセシビリティ検証必須」というプロセスが組織の標準化され、装飾優先の無責任な改修が防止される
- **採用・営業チームへの信頼度向上**：「非エンジニアにも使いやすい設計」という約束が実装された状態でリリース可能

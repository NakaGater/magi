ローカルリポジトリの編集
Webアプリ（ブラウザ）から直接ローカルファイルを触ることはブラウザのセキュリティ上できません。ただし、Magiは既にサーバーデーモンが常駐する設計なので、このサーバーがローカルファイルを操作する構成で解決できます。今の計画書の設計そのままです。

ブラウザ → Magi Server (localhost) → ローカルのGitリポジトリ
つまりユーザーが自分のマシンでmagi initすれば、ローカルサーバーがファイルの読み書きとGit操作を行い、ブラウザはそのサーバーとやり取りするだけです。

リモートリポジトリの編集
こちらは2つのアプローチがあります。

**A. クラウドホスト型（SaaS）**として提供する場合、Magiサーバーをクラウドに置き、GitHub/GitLab APIでリモートリポジトリをclone→編集→push/PR作成します。ユーザーはブラウザだけで使えて、ローカルに何もインストール不要です。ただしクラウド上でLSPやビルドを動かすサンドボックス環境が必要になり、インフラの複雑さとコストが大きく上がります。

B. ローカル実行のまま、リモートリポジトリを対象にする場合は、Magiサーバーがgit cloneでリモートを取得し、ローカルで編集して、完了後にgit pushまたはPR作成する流れです。インフラは不要ですが「ローカルにMagiを入れる」ステップは残ります。

この2点を聞かせてもらえると、具体的なアーキテクチャの検討に入れます。SaaSにする場合はインフラ設計がかなり変わるので、方向性は早めに決めておきたいところです。

1. プロジェクト単位で常に参照する情報（永続コンテキスト）

.magi/context/にMarkdownやYAMLを置いておく方法です。Magiが議論を開始するたびに自動で読み込みます。

.magi/context/
  ├── product.md          ← プロダクトの概要、ミッション、ターゲットユーザー
  ├── competitors.md      ← 競合分析、差別化ポイント

  └── glossary.md         ← ドメイン用語の定義
Claude CodeのCLAUDE.mdと同じ発想ですが、ロールごとに関連する情報を優先的に参照させます。PMにはproduct.mdとcompetitors.md、PDにはdesign-system.md、Devにはtech-stack.mdを重点的に。

2. タスク投入時に添付する情報（タスクコンテキスト）

これはFigmaのような視覚的なデザインシステムではなく、AIロールが読んで理解するためのテキスト仕様です。PDロールがHTMLプロトタイプを生成する時に「ボタンは角丸6px、高さ36px、Primary は黒背景」と参照し、議論で「影は使わない方針なのでドロップシャドウは不採用」と判断できるようになります。

ポイントは「やらないこと」まで明記していることです。AIは制約がないと何でもやろうとするので、グラデーション禁止、シャドウ禁止、ネスト禁止といったルールがあると、生成物の一貫性が上がります。

実際にはプロジェクトごとに内容は全然違います。この例はミニマル系のWebアプリを想定していますが、ポップなゲームアプリならカラーパレットもトーンも全く変わります。チームが.magi/context/design-system.mdを自分たちのプロジェクトに合わせて書くだけです。
